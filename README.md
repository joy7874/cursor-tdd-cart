# Cart Discount TDD Practice

## 목적

이 프로젝트는 **장바구니 할인 계산 로직**을 TDD(Test-Driven Development) 방식으로 구현하는 연습 프로젝트입니다.

도메인 로직은 `src/cart.py`의 Entity 계층에 위치합니다. 모든 테스트와 구현은 **계약 ID**(`INV-*`, `E-*`)를 기준으로 작성·추적합니다. 계약 ID는 테스트와 구현을 잇는 **추적의 못**이며, **ID에 없는 동작은 구현하지 않습니다.**

현재 단계는 **RED 준비 단계**입니다. 아직 구현이 완료된 상태가 아닙니다.

## 핵심 원칙

- **ID에 없는 동작은 만들지 않는다.** 계약 표에 없는 할인·예외·기능은 추가하지 않습니다.
- **테스트가 먼저다.** 구현보다 실패하는 테스트가 항상 앞섭니다.
- **RED → GREEN → REFACTOR** 순서를 따릅니다.
  - RED: `tests/`만 수정, 실패 테스트 작성
  - GREEN: `src/`에 해당 ID를 만족하는 최소 구현
  - REFACTOR: 전부 통과한 뒤에만 구조 개선
- **과잉 구현을 금지한다.** 요청·계약에 없는 기능을 미리 만들지 않습니다.

## 계약 ID 목록

| ID    | 계약(불변식 / 에러)                                                 | 근거 레벨 | 계층        |
| ----- | ------------------------------------------------------------ | ----- | --------- |
| INV-1 | `subtotal(items) == Σ(price × qty)`                          | —     | Entity    |
| INV-2 | `amount ≥ 50000 → round(amount×0.9)` / `< 50000 → 그대로` 경계 포함 | L1    | Entity    |
| INV-3 | `final = 문턱할인 적용 후, VIP면 round(×0.95)`. 순서 문턱→VIP 고정         | L2    | Entity    |
| INV-4 | 모든 입력에서 `0 ≤ final_total ≤ subtotal`. 할인은 금액을 늘리지 않는다        | L3    | Entity    |
| E-1   | `items is None → TypeError`                                  | L0    | Boundary* |
| E-2   | `price` 또는 `qty`가 음수 → `ValueError`, 인덱스 포함                  | L0    | Boundary* |

## 계약 ID 설명

### INV-1

장바구니 소계는 각 품목의 `price × qty`를 모두 더한 값과 같아야 합니다.

### INV-2

금액이 50,000원 이상이면 10% 할인(`round(amount × 0.9)`)을 적용하고, 50,000원 미만이면 원래 금액을 그대로 반환합니다. 경계값 50,000원은 할인이 적용되는 쪽(이상)에 포함됩니다.

### INV-3

최종 금액은 문턱 할인을 먼저 적용한 뒤, VIP 고객이면 `round(× 0.95)`를 추가 적용합니다. 적용 순서는 **문턱 → VIP**로 고정됩니다.

### INV-4

어떤 입력에서도 최종 결제 금액은 0 이상이며 소계 이하이어야 합니다. 할인은 절대 금액을 늘리지 않습니다.

### E-1

`items`가 `None`이면 `TypeError`를 발생시킵니다.

### E-2

`price` 또는 `qty`가 음수이면 품목 인덱스를 포함한 `ValueError`를 발생시킵니다.

## 계층 의미

### Entity

`src/cart.py`에 위치하는 **순수 도메인 계산 로직** 계층입니다. Flask 등 Boundary 프레임워크를 import하지 않습니다. `INV-*` 계약은 이 계층의 불변식을 정의합니다.

### Boundary*

`E-*` 계약은 입력 검증 경계에 가까운 규칙입니다. 본 실습에서는 별도의 웹/API 계층 없이 **도메인 함수 진입점**에서 입력을 검증합니다. `items is None`, 음수 `price`/`qty` 등은 함수 호출 시점에 거부됩니다.

## TDD 진행 순서

1. **RED** — 계약 ID별로 실패하는 테스트를 작성합니다. 이 단계에서는 `tests/`만 수정하고 `src/`는 건드리지 않습니다.
2. **GREEN** — 해당 계약 ID를 만족하는 **최소 구현**만 `src/cart.py`에 추가합니다. 구현 줄에는 충족한 계약 ID를 주석으로 표기합니다.
3. **REFACTOR** — 모든 테스트가 통과한 상태에서만 구조를 개선합니다. 리팩터 전후로 `pytest -q`로 동작 불변을 확인합니다.

## 테스트 실행

```bash
pytest -q
```

`-q`는 quiet mode로, 테스트 실행 결과를 간결하게 출력합니다.

## 구현 금지 사항

- 할인 정책 추가 금지 (계약 표에 없는 새 할인 규칙)
- 쿠폰, 세금, 배송비, 포인트 기능 추가 금지
- ID에 없는 예외 처리 추가 금지
- UI, CLI, DB, API 코드 추가 금지
